# Civic Radar: Community Hero Platform Audit (ProblemV1.md)

This audit report compiles all existing issues, software architecture limitations, missing features, user experience gaps, and optimization vectors identified across the codebase (`App.tsx`, `MapCanvas.tsx`, `dbService.ts`, `server.ts`, and core React components).

---

## 1. Core Architectural & Persistence Gaps

### ⚠️ Local-Only Fallback Loop (Storage Volatility)
* **Problem**: When Firebase authentication and Firestore are unconfigured, the application gracefully falls back to standard `localStorage` via `dbService.ts`. While this ensures the app does not crash, any browser cache clear or cookie sweep results in the complete loss of:
  * User-reported hazards and photos.
  * Discussion threads and neighborhood comment logs.
  * Acquired civic points, badges, and leaderboard standings.
* **Impact**: Decreases long-term user retention and dampens the "Hero" incentive loop.
* **Improvement**: Introduce an automated "Backup/Export to JSON" utility or prompt users to link their cloud profiles to persist data across multiple devices.

### ⚠️ Local Profile Static Hardcoding
* **Problem**: In `dbService.ts`, the profile is initialized to a static placeholder profile: **Jane Civic** (`uid: "user-local-hero"`, `points: 25`, avatar `Jane`).
* **Impact**: There is absolutely no in-app profile editor. Users are locked into the "Jane Civic" persona without the ability to change:
  * Display name or email.
  * Avatar picture.
  * Civic neighborhood classification (Ward/District).
* **Improvement**: Build a compact profile customizer dialog or settings tab in the left panel.

## 2. API & Geocoding Constraints

### 🗺️ Public Nominatim Geocoding Rate-Limits
* **Problem**: The live interactive map (`MapCanvas.tsx`) performs real-time reverse-geocoding requests directly to the open-source OpenStreetMap **Nominatim API**:
  `https://nominatim.openstreetmap.org/reverse?format=jsonv2&lat={lat}&lon={lng}`
* **Impact**: The Nominatim API is a shared, free service that enforces strict rate limits. High traffic or rapid double-clicking on the map canvas can cause:
  * `HTTP 429 Too Many Requests` response errors.
  * Slow geocoding response lag, forcing the app to fall back to a mock simulated intersect title.
* **Improvement**: Implement client-side debounce layers, caching of previously requested lat/lng pairs, or transition to a premium geocoding service with proxy caching on the backend.

### 📡 Server-Side Gemini API Demarcation
* **Problem**: If the `GEMINI_API_KEY` is not present in the backend's environmental configurations, the backend routing proxies gracefully route to "Demo Mode" mock predictions.
* **Impact**: While it's highly functional for instant evaluations, users might not immediately realize that the categorization is static and generated by an offline fallback rather than the live Gemini 3.5 Flash model.
* **Improvement**: Add a subtle, high-contrast banner indicator within the Report creator or active post details if the backend is currently serving "Demo Mode" AI mock responses.

## 3. UI, UX & Interaction Flaws

### 🪟 Hardcodedproof-of-Resolution Image
* **Problem**: When a citizen marks a hazard as "Resolved", the system automatically links a beautiful static stock photo of a paved street:
  `https://images.unsplash.com/photo-1541888946425-d81bb19240f5`
* **Impact**: Neighbors cannot submit their *actual* visual proof of repair. This opens up avenues for system gaming where users can claim 50 civic points easily by clicking "Resolve" without executing any neighborhood cleanup.
* **Improvement**: Add an optional photo upload input inside the "Mark Resolved" dialog so users can attach real visual proof.

### 📱 Tablet & Mobile Responsive Navigation Density
* **Problem**: The side navigation tabs ("Radar", "Active Feed", "Analytics", "Leaderboard") collapse nicely, but the main spatial view is extremely dense on mobile screens. The Leaflet map height combined with feed cards forces extensive page scrolling.
* **Impact**: Users on small touch-screens may struggle to inspect the map while referencing the active feed or adding comments.
* **Improvement**: Utilize a sticky split-screen view on mobile viewport sizes (Map on top half, dynamic tabs list on bottom half) or apply touch-gesture sliders.

### 🎨 Low Category Tag Contrast
* **Problem**: The light theme badges use low-contrast pastel backgrounds (e.g., yellow background `bg-amber-100` with yellow text `text-amber-700` for `Pothole` tags).
* **Impact**: Fails standard **WCAG AAA** accessibility criteria for vision-impaired users under bright outdoor sunlight conditions (where municipal reporting is most likely to occur).
* **Improvement**: Boost text density or substitute with high-contrast icon combinations.

## 4. Feature Gaps (Functional Enhancements)

| Missing Feature | Description & Utility | Priority |
| :--- | :--- | :--- |
| **DPW PDF Export** | An option to generate a structured PDF petition containing the geolocated pin, verified photos, and community upvotes to directly email or print for the local **Department of Public Works (DPW)**. | **High** |
| **Proximity Filters** | Filter incidents by radius (e.g., "Within 500m", "Within 1km") using the browser's real-time Geolocation API instead of matching purely via title/description keywords. | **Medium** |
| **Official Authorities Portal** | A secondary user role classification (e.g., "City Worker", "Municipal Inspector") with the power to issue official updates or lock comments once an issue is scheduled for professional repair. | **Medium** |
| **Notification History Hub** | A panel listing historic alerts (e.g., "A pothole was reported 200m near your saved address" or "A leak you verified was marked resolved!"). | **Low** |
| **Upvote Throttling & Anti-Abuse** | Security gates to prevent a single profile from upvoting their own reports or scripting endless upvotes to game the global Leaderboard standings. | **Low** |
